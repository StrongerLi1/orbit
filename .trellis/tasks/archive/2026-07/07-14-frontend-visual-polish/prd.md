# 优化前端视觉体验

## Goal

在不改变现有功能、权限和数据行为的前提下，重新设计 Orbit 的前端界面。成品应是一套温暖、有层次、适合长期使用的个人工作台：保留少量书房气质，但不做成密集细线和大段留白的编辑排版样张，也不套用常见 AI 产品模板。

## Background

- Orbit 是管理收藏、计划、待办、摘录、网盘搜索和 Hermes 的单页个人工作台。
- 前端由 `public/index.html`、`public/styles.css` 和 `public/app.js` 组成；PlayCaptcha 是嵌入现有页面的独立 React island。
- 页面交互依赖现有元素 `id`、`data-*` 属性和 `active`、`open`、`done` 等状态类。
- 用户允许大幅调整视觉、DOM 布局和纯展示文案，不要求延续当前暖灰、深色侧栏、橙色强调色和卡片墙结构。

## Requirements

### R1. 功能兼容

- 保留全部页面入口、表单字段、快捷键、权限可见性、增删改查、搜索、打卡、聊天和管理流程。
- 保留 JS 使用的元素 `id`、`data-*` 属性和状态类语义；视觉改造不得改变 API 请求、状态管理或业务判断。

### R2. 视觉方向

- 采用现代书房式个人工作台：深橄榄导航、暖灰背景、统一的亚麻操作色与鼠尾草软色块、适度圆角和轻量阴影。
- 标题与摘录可以使用少量衬线字体，普通界面仍使用系统无衬线字体；用色块、留白和有限卡片表达层级，避免密集细线和过度极简。
- 同一套视觉语言必须覆盖登录、应用框架、业务页面、聊天、管理页、弹窗、空状态和反馈状态。

### R3. 去除 AI 模板感

- 不使用无目的渐变、玻璃拟态、装饰性发光、泛用星芒/机器人符号、过度胶囊按钮或同质化卡片堆叠。
- 装饰必须帮助理解信息结构、状态或操作，不以“科技感”和“高级感”为独立目标。
- 删除英文眉题、泛化励志句、营销式措辞和机械的中英混排。

### R4. 文案

- 允许重写纯展示文案、空状态和说明文字。
- 功能名称、字段含义、操作结果、风险提示和权限说明必须保持准确。
- 新文案使用自然、具体、简短的中文，并明确当前状态或下一步操作。

### R5. 响应式与可访问性

- 桌面端应支持常见 1280px 以上宽度；窄屏应覆盖 390px 级别手机视口。
- 不得出现主要内容横向溢出、关键操作遮挡或过小的触控目标。
- 键盘焦点、hover、active、disabled、错误和成功状态必须清楚可辨，正文与控件文字保持可读对比度。

### R6. 技术约束

- 可重构现有 HTML 和 CSS；只在展示文案或视觉状态确有需要时修改 `public/app.js`。
- 不新增前端框架、组件库、图标库或运行时依赖。
- 不修改后端 API、数据库、权限模型和业务规则。

## Acceptance Criteria

- [x] AC1: 现有自动化测试与 `public/app.js` 语法检查全部通过。
- [x] AC2: 全部 JS 依赖的 `id`、`data-*` 属性和状态类仍存在并可驱动原有交互。
- [x] AC3: 登录、概览、收藏、摘录、计划、待办、网盘、Hermes 聊天、Hermes 管理、用户管理和通用弹窗使用统一的新视觉系统。
- [x] AC4: 页面层级不依赖渐变、玻璃拟态、装饰性发光、泛用星芒、过度胶囊化或同质化卡片墙。
- [x] AC5: 英文装饰标签和泛化励志文案已清理；保留文案能说明功能、状态或操作。
- [x] AC6: 在 1440×1000 与 390×844 视口完成关键页面视觉核验，无明显遮挡、溢出或不可点击区域。
- [x] AC7: 可交互控件具有一致且清楚的 hover、active、focus-visible 和 disabled 反馈。
- [x] AC8: 生产代码未新增运行时依赖，业务 API 和权限行为无变更。

## Out of Scope

- 新增或删除产品功能与页面。
- 增加深色模式、主题切换或用户自定义外观。
- 修改 PlayCaptcha 内部交互、后端服务、数据库或部署配置。
