# 待办和计划按用户隔离

## 目标与用户价值

让每个登录用户只看到、创建、修改和删除自己的待办与计划，避免不同账号之间共享个人工作内容。

## 背景与已确认事实

- 认证上下文已提供稳定的当前用户 ID、用户名和权限；后端通用内容接口通过 `require_permission` 获取当前用户。
- MySQL `todos` 和 `plans` 表当前没有用户归属字段。
- `backend.repository.list_items()`、`get_item()`、`create_item()`、`update_item()`、`delete_item()` 对待办和计划使用全表/ID 查询，没有用户过滤或归属校验。
- `backend/main.py` 的 `/api/{collection}` 通用路由目前对待办和计划沿用共享内容逻辑；浏览器 `public/app.js` 与 Android 客户端分别通过 `/api/todos`、`/api/plans` 加载并执行 CRUD。
- 启动数据库初始化和旧 JSON 导入会生成无归属的待办、计划种子数据；已有数据需要兼容迁移。

## 需求

- 待办和计划记录必须绑定创建时的认证用户 ID。
- 列表、单条查询、更新、删除都必须按当前用户归属过滤，不能依赖客户端隐藏入口实现安全隔离。
- 浏览器和 Android 客户端继续使用现有 API 路径，但只显示当前用户自己的数据。
- 管理员也只查看和维护自己的待办、计划，不保留管理员查看或管理他人记录的特殊分支。
- 历史上没有归属的待办、计划全部绑定到配置的管理员账号；迁移必须幂等且不丢失内容。

## 产品决策

- 归属判断严格使用认证用户 ID，不使用用户名或客户端传入字段。
- 历史数据优先绑定配置的管理员账号；若配置账号不存在，使用首个管理员账号。
- API 响应继续保持待办和计划现有字段，不为本需求暴露内部用户 ID。
- 本次只隔离待办和计划，不改变书签、文件夹、摘录、书库等其他业务数据的共享语义。

## 验收标准

- **AC-1：** 用户 A 创建的待办/计划不会出现在用户 B 的列表、仪表盘或 Android 客户端中；管理员账号同样只看到自己归属的记录。
- **AC-2：** 用户 B 不能通过已知 ID 查询、修改或删除用户 A 的待办/计划；接口返回拒绝结果且原记录不变。
- **AC-3：** 用户只能创建和维护自己的待办/计划；浏览器和 Android 的新增、完成/重开、计划打卡、删除和清空已完成流程正常。
- **AC-4：** 旧数据库启动升级会幂等添加归属字段并把无归属记录绑定管理员；旧 JSON 导入数据也最终归属管理员，不重复创建或丢失数据。
- **AC-5：** 现有待办/计划字段、日期统计、计划完成次数和页面交互保持不变。
- **AC-6：** 后端自检、浏览器 JavaScript 语法检查、Android 单元测试、lint 和 debug 构建通过。

## 非目标

- 不新增角色或权限类型。
- 不允许客户端指定或修改记录归属。
- 不把待办或计划改造成管理员可跨用户审计的后台功能。
- 不扩展到其他 collection 的用户隔离。
