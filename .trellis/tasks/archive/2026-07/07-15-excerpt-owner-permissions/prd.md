# 摘录记录用户与个人权限

## 目标与用户价值

为摘录记录写下摘录的用户，让共享摘录可追溯；新摘录保存当前登录用户，普通用户可以维护自己的摘录，管理员可以维护全部摘录。

## 当前已知事实

- 摘录存储在 MySQL `excerpts` 表，目前没有归属用户字段。
- 后端使用通用 `/api/excerpts` 增删改接口；当前所有拥有 `content:write` 权限的用户都可以修改或删除全部摘录。
- 浏览器端已有摘录列表、添加和删除入口，但没有编辑入口；Android 端已有摘录列表、添加和删除入口。
- 项目现有认证模型提供当前用户 ID、用户名和 admin/user 角色。
- 需要兼容已有数据，并将已有摘录归属到 `admin`。

## 需求

- 摘录数据包含摘录人信息，并在浏览器和 Android 客户端展示摘录人的用户名。
- 创建摘录时由服务端记录当前登录用户 UID，客户端不能伪造摘录人；用户名只用于展示。
- 普通用户可编辑和删除自己的摘录。
- 管理员可编辑和删除全部摘录；普通用户不能编辑或删除他人的摘录。
- 保持所有已登录用户可读取摘录的现有共享行为。
- 浏览器和 Android 客户端都提供摘录编辑入口，并仅展示当前用户可执行的编辑/删除操作。

## 验收标准

- **AC-1**：新建摘录后，API、浏览器端和 Android 端都能看到当前登录用户的摘录人用户名；请求体中的伪造摘录人字段不会改变服务端记录。
- **AC-2**：旧数据读取正常，且未记录摘录人的历史数据按 `admin` 展示和授权。
- **AC-3**：普通用户可以编辑和删除自己的摘录；非摘录人操作他人摘录返回拒绝结果，数据不变。
- **AC-4**：管理员可以编辑和删除任意摘录。
- **AC-5**：所有用户仍能读取全部摘录，摘录人的展示字段不会泄露不必要的用户内部 ID。
- **AC-6**：浏览器和 Android 端的编辑表单会回填现有摘录内容、作者、出处、日期和备注；保存或删除后列表状态正确刷新。
- **AC-7**：数据库启动升级幂等，旧数据库和 JSON 导入路径都能为摘录设置默认 `admin` 归属；相关后端、前端语法和 Android 检查通过。

## 产品决策

- 管理员保留全部摘录的编辑和删除权限；普通用户只拥有自己摘录的写权限。
- 普通用户的归属判断严格使用认证用户 UID，不使用用户名或客户端传入的摘录人字段。
- 摘录继续对所有已登录用户共享可读；归属只约束编辑和删除，不改变现有列表和概览展示范围。
- 本次同时更新浏览器客户端和 Android 客户端，避免两个官方客户端产生不同的权限体验。

## 非目标

- 不给收藏、待办、计划、收藏夹等其他业务数据增加用户归属。
- 不把摘录改为按用户隔离的私有列表。
- 不新增角色、权限类型或管理员页面。
- 不允许用户手工选择或修改摘录人。
